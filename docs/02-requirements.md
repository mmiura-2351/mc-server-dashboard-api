# v2 要件定義

v1 の破綻分析 (`01-current-issues.md`) を踏まえ、v2 が満たすべき機能/非機能要件を整理します。
本ドキュメントは **アーキテクチャ非依存の「何を満たすか」** を記述し、
具体的な実装方針は `03-architecture-direction.md` に分離します。

## 1. 用語

| 用語 | 定義 |
|------|------|
| Organization | リソース分離の単位となる組織。サーバー/バックアップ/グループはいずれかの Organization に属する |
| User | 認証主体。複数の Organization に所属し得る |
| Member | User と Organization の紐付け。RoleTemplate と個別 Permission を持つ |
| Server | 管理対象の Minecraft サーバーインスタンス |
| Runner | Minecraft サーバープロセスを実際に動かす実行基盤 (コンテナ/ホスト) |
| Job | 起動/停止/バックアップ/復元などの非同期タスク |

## 2. スコープ

### 本プロジェクトが提供するもの
- Minecraft サーバーのライフサイクル管理 API (作成/起動/停止/設定/バックアップ/復元)
- Organization 単位のアクセス制御 (メンバー/権限)
- Runner (実行基盤) に対する抽象化された制御インタフェース
- リアルタイムのログ/ステータス配信
- 監査ログ

### 明確にスコープ外
- Minecraft クライアント側機能
- サーバーホスティング業務そのもの (Runner の物理調達・課金ゲートウェイ等)
- フロントエンド UI (別リポジトリ)
- MOD/プラグインそのものの配布
- 多言語対応 (UI は単一言語で実装する)

## 3. 機能要件

### 3.1 Organization / ユーザー管理
- **FR-O-1** Organization を作成/削除できる。削除は配下のサーバーがすべて削除済みであることを前提とし、サーバーが残っている場合は削除不可とする（誤削除防止）
- **FR-O-2** 1 ユーザーは複数の Organization に所属でき、Organization ごとに異なるロールを持てる
- **FR-O-3** メンバーに対して **機能単位** のパーミッションを付与できる (例: `server.start`, `server.delete`, `backup.restore`)
- **FR-O-4** パーミッションは Role テンプレート (`owner`, `admin`, `operator`, `viewer`) からプリセット適用でき、かつ個別に上書きできる
- **FR-O-5** API トークン (PAT) をメンバー権限のサブセットとして発行できる (Phase 2)
- **FR-O-6** 招待フロー (メール等) でユーザーを Organization に追加できる (Phase 2)

### 3.2 サーバー管理
- **FR-S-1** Organization 内にサーバーを作成でき、作成時に Runner 種別を選択する
- **FR-S-2** サーバーは起動/停止/再起動/強制停止ができ、状態遷移は監査ログに記録される
- **FR-S-3** サーバーは Minecraft バージョン/サーバータイプ (vanilla/paper/forge/fabric 等) を指定できる
- **FR-S-4** `server.properties` など設定ファイルを API 経由で読み書きできる
- **FR-S-5** サーバーのポートは **Runner が自動割り当て** し、外部公開ポート/内部ポートを API が返却する
- **FR-S-6** 同一 Organization 内で **任意数のサーバーを同時実行** できる (ポート/リソース競合を Runner が解決する)
- **FR-S-7** サーバーごとに CPU/メモリ/ディスクのリソース上限を設定できる

### 3.3 Runner (実行基盤)
- **FR-R-1** 実行基盤は **差し替え可能なプラガブル構造** とし、MVP の主ターゲットはコンテナ Runner (Docker/Podman) とする。ホスト Runner (直接プロセス起動) はオプション実装として許容する
- **FR-R-2** API Core を再起動しても、稼働中の Minecraft サーバーは継続稼働し、状態再同期で追跡を復元できる
- **FR-R-3** Runner と API Core は認証付きチャネルで通信し、Runner は外部から直接操作できない

### 3.4 ジョブ/非同期処理
- **FR-J-1** 起動/停止/バックアップ/復元はすべて **非同期ジョブ** として扱い、ジョブ ID を返却する
- **FR-J-2** ジョブはキューに永続化され、ワーカー/Runner の再起動をまたいで完了する
- **FR-J-3** ジョブは状態 (`queued` / `running` / `succeeded` / `failed` / `cancelled`) を持ち、進捗を取得できる
- **FR-J-4** 失敗ジョブは手動リトライできる

### 3.5 バックアップ/復元
- **FR-B-1** 手動/スケジュールでのバックアップが可能 (時刻ベースのスケジュール式)
- **FR-B-2** バックアップはローカルまたは S3 互換オブジェクトストレージに保存できる (設定で切り替え可能)
- **FR-B-3** 復元は既存サーバーへの上書き、または新規サーバー作成として選択できる
- **FR-B-4** 復元先は同一 Organization 内のサーバーに限定される

### 3.6 リアルタイム配信
- **FR-W-1** サーバーログ/コンソール/状態変化を WebSocket で配信する
- **FR-W-2** 配信は認可チェック付きで、購読者は該当サーバーに対する閲覧権限を持つ必要がある
- **FR-W-3** コンソールコマンドは権限のあるユーザーのみ送信可能で、監査ログに記録される

### 3.7 ファイル管理
- **FR-F-1** Runner 上のサーバーディレクトリに対し、API 経由でファイル一覧/読み書き/アップロード/ダウンロードができる
- **FR-F-2** パストラバーサル防止を Runner 側で強制する
- **FR-F-3** 編集履歴を保持し、ロールバック可能

### 3.8 グループ (プレイヤー集合)
- **FR-G-1** OP/ホワイトリストを Organization 内で共有可能な「グループ」として定義できる
- **FR-G-2** グループはサーバーに attach でき、attach 時に対応 JSON ファイルが生成・更新される
- **FR-G-3** プレイヤーの UUID/名前解決は Mojang API への問い合わせを中央でキャッシュする

### 3.9 監査/可観測性
- **FR-A-1** すべての変更系 API 呼び出しは監査ログに残る (who / when / what / from)
- **FR-A-2** 監査ログは Organization の owner/admin が閲覧でき、日付/ユーザー/リソース種別でフィルタできる

## 4. 非機能要件

### 4.1 スケーラビリティ
- **NFR-SC-1** API Core はステートレスとし、複数レプリカで水平スケールできる
- **NFR-SC-2** Runner は水平方向に追加可能で、新規 Runner の追加は無停止で行える

### 4.2 信頼性
- **NFR-RL-1** API Core の再起動によって Minecraft サーバーは停止しない
- **NFR-RL-2** ジョブは **at-least-once** で実行され、ジョブ側で冪等性を担保する

### 4.3 セキュリティ
- **NFR-SE-1** すべての API は認証済みで、Organization 境界を越えたリソース参照を禁止する
- **NFR-SE-2** ファイル操作は Runner のサンドボックス外へ出られない
- **NFR-SE-3** 秘匿情報 (RCON パスワード、DB 認証情報) は Secret 管理基盤で扱う
- **NFR-SE-4** 依存ライブラリの脆弱性スキャンと SBOM 出力を CI で自動化する

### 4.4 性能
- **NFR-PF-1** 状態問い合わせ API (ログ/ステータス以外) は p95 < 300ms
- **NFR-PF-2** サーバー起動ジョブはキュー投入から Runner 起動開始まで p95 < 5s

### 4.5 可観測性/運用性
- **NFR-OP-1** 構造化ログ (JSON) を標準出力に出し、trace_id を全レイヤで貫通させる
- **NFR-OP-2** OpenTelemetry 互換のトレーシングを API/ワーカー/Runner に入れる
- **NFR-OP-3** デプロイはコンテナイメージ経由で、ヘルスチェック/readiness を提供する

### 4.6 保守性
- **NFR-MT-1** ドメイン層は実行基盤実装 (Docker/ホスト fork) に依存しない
- **NFR-MT-2** DB マイグレーションはツール管理 (Alembic 等) とする
- **NFR-MT-3** 主要ドメインは外部記述可能な仕様 (OpenAPI, AsyncAPI) を CI で公開する

## 5. 制約条件

- **C-1** 既存フロントエンド (`mc-server-dashboard-ui`) との **API 互換は維持しない**。v2 は新規 API 契約で設計する
- **C-2** v1 のデータ (サーバーディレクトリ、DB) からの移行は MVP では対象外とする
- **C-3** 初期ターゲット Runner はコンテナ (Docker/Podman)。K8s 対応は MVP 直後のフェーズ
- **C-4** Python で実装する方針は継続するが、言語選定は設計レビューで再確認する

## 6. 優先度 (MVP → 後続)

### MVP (Phase 1)
- Organization/メンバー/機能別パーミッション (3.1)
- サーバー作成/起動/停止/設定編集 (3.2)
- コンテナ Runner (3.3 の一部)
- ジョブキュー/非同期実行 (3.4)
- 手動バックアップ (3.5 の一部)
- WebSocket ログ配信 (3.6)
- 監査ログ (3.9 の一部)

### Phase 2
- スケジュールバックアップ
- 複数 Runner 冗長化 (FR-R-2)
- Kubernetes Runner
- API トークン (PAT) / 招待フロー (FR-O-5, FR-O-6)

### Phase 3 以降
- リージョン分散、マルチクラウド
- メトリクス公開 (Prometheus 互換)

## 7. オープン事項

合意形成が必要で、要件 FIX 前に決めたい項目。

- **OQ-1** Runner 認証方式 (mTLS / OIDC / 単純トークン) のどれを採用するか
- **OQ-2** ジョブキューの実装選定 (Redis / Postgres-based / NATS / RabbitMQ)
- **OQ-3** DB は引き続き SQLite で MVP を作るか、最初から PostgreSQL を前提にするか (`03-architecture-direction.md` では PostgreSQL を推奨。SQLite はマルチインスタンス化に不向きなため)
- **OQ-4** v1 の既存ユーザー/サーバーを v2 に移行する必要があるか
