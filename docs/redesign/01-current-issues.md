# v1 現行設計の破綻整理

設計をまとめずに機能追加を続けた結果、以下のレイヤで構造的な問題が発生しています。
各項目は「後からのリファクタで解消が困難」であるため、v2 では要件レベルから作り直します。

## 1. ユーザー/権限モデルの硬直化

### 現状
- `app/users/models.py` の `Role` は `admin / operator / user` の 3 値 enum
- リソース所有は `Server.owner_id` 等の単一外部キーで表現
- 承認フラグ (`is_approved`) と有効フラグ (`is_active`) が直接モデルに埋め込まれ、権限チェックがサービス層に散在
- `AuthorizationService` が各ドメインを個別に if 分岐

### 破綻している点
- **機能単位の粒度で ON/OFF できない**: 「このユーザーはバックアップ作成は可能だが復元は不可」のような制御ができない
- **組織/テナント概念がない**: 複数ユーザーを束ねる単位が存在せず、共同運用の表現が不可能
- **所有者の委譲・共有ができない**: 単一 FK のため「複数人で 1 サーバーを運用」が表現できない
- **ロール拡張が破壊的**: enum 値追加には全権限分岐の見直しが必要

### 影響度
- 高。認可モデルはほぼ全ドメインに横断するため、後から差し替えるのは事実上不可能。

## 2. サーバープロセスが API と同一ホストで動作

### 現状
- `app/services/minecraft_server.py` が `os.fork()` を用いた double-fork でローカル Java プロセスを起動
- `./servers/{id}/` 配下のファイルを API プロセスが直接読み書き
- PID ファイル (`minecraft_server.pid`) をローカルファイルシステム上で管理

### 破綻している点
- **水平スケール不可能**: API サーバーを複数台に分けると、サーバープロセスをどのホストに起動したか追跡不能
- **リソース競合**: API の CPU/メモリと Minecraft サーバーの CPU/メモリが同じプールを食い合う
- **障害波及**: API プロセスがクラッシュするとローカル監視が失われ、また Minecraft サーバー側の不具合 (OOM 等) が API ホストに波及
- **デプロイ不可逆性**: API を再起動すると稼働中サーバーとの PID 同期が必要で、運用が脆い

### 影響度
- 高。プロセス実行モデルはアーキテクチャ全体の前提であるため、既存コードの大半が書き換え対象になる。

## 3. 同時実行性の欠如

### 現状
- ポート (`Server.port`) はユーザーが手動設定、重複は DB の UNIQUE 制約等でも保証されていない
- JAR キャッシュ、ファイルロック、RCON ポートなど共有リソースの排他制御が場当たり的
- バックアップ・起動・停止が同一プロセス内 asyncio タスクで実行されるため、ひとつの長時間処理が他をブロック

### 破綻している点
- **ポート/リソースの動的割り当てができない**: 10 台以上を同時運用しようとすると手作業での競合解消が必要
- **バックアップ/起動の並列実行が怖い**: 排他制御が分散しているため、並列実行時の一貫性を保証できない
- **キューイング基盤がない**: ジョブが API プロセスに直結しており、再起動で失われる

### 影響度
- 中〜高。ビジネス要件として「複数テナントが同時に稼働する」を前提にする場合、基盤ごと差し替える必要がある。

## 4. 実行環境の固定化

### 現状
- Java 実行バイナリパスを `.env` で直指定 (`JAVA_8_PATH` ... `JAVA_21_PATH`)
- `JavaCompatibilityService` がホスト上の Java を検出して使う前提
- 実行は fork + exec で、プロセスは API ホストの OS/libc/カーネルに完全依存

### 破綻している点
- **環境差異が吸収できない**: mod が要求するネイティブライブラリ、glibc バージョン、追加ツール (Python, Node, GPU ドライバ等) を個別に用意する手段がない
- **バージョン混在が辛い**: Java 8/17/21 を同時に運用するには物理的にホストへインストールする必要がある
- **再現性が低い**: 開発環境と本番環境で挙動が異なる原因になる
- **`screen` 相当のセッション管理に依存**: TTY の有無や signal handling をホストに委ねており、コンテナ化/サンドボックス化の障害になる

### 影響度
- 高。コンテナ/VM/リモートワーカーに逃がすには、実行モデル自体を差し替える必要がある。

## 5. テナント分離/グループ単位のサーバー分離の欠如

### 現状
- `Group` モデルは OP/ホワイトリストのプレイヤー集合を表すのみ (認可単位ではない)
- サーバーは `owner_id` により個人に紐づくだけで、「組織」「チーム」「環境 (dev/prod)」の概念がない
- ファイルシステム配置 `./servers/{id}/` はフラットで、テナント境界が存在しない

### 破綻している点
- **共同運用ができない**: 1 つのサーバーを複数メンバーで運用する要件を満たせない
- **テナント単位でのクォータ/課金/監査ができない**: 利用量集計が個人単位に限定される
- **データ削除境界が曖昧**: テナント退会時にどのリソースを消すべきか一意に決まらない

### 影響度
- 高。テナント境界はデータモデル全体の主軸となるため、後付けでは破壊的変更になる。

## 6. 副次的に発生している構造上の負債

優先度は低いが、設計を改める際には同時に解消したい項目。

- `app/services/` に 20 以上のサービスが平置きされ、ドメイン境界が曖昧
- `app/core/visibility*.py` が後付けの横断的関心事としてコアに混入
- `FileEditHistory` などドメイン特化の監査が、横断監査 (`AuditLog`) と二重化
- WebSocket サービスが `MinecraftServerManager` に直接依存し、実行基盤の差し替えを妨げている
- バージョン情報取得が実行時外部 API 依存 (`REFACTOR_PLAN.md` で別途議論済み)
- SQLite 固定前提の設計 (マルチテナント化時にスケールしない)

## まとめ

1〜5 は **v1 の延長では解消不能** な構造問題であるため、v2 では以下を前提条件とします。

- 認可モデルを先に設計する (機能別権限 × テナント)
- サーバー実行基盤を API から分離する (ワーカー/エージェントモデル)
- 実行環境をサンドボックス化する (コンテナ/仮想化前提)
- マルチテナントをデータモデルの主軸に据える
- ジョブ実行は非同期キュー基盤に載せる
